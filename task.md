Цель: спроектировать, реализовать и защитить ML-систему, устойчивую к качеству данных, изменению распределений и эксплуатационным ограничениям.
Чекпоинты
Чекпоинт 1. Постановка задачи и первичное проектирование
1. Выбор темы и постановка задачи
Выбранная тема (зафиксировать в журнале).
Формализация задачи (тип ML-задачи; объект и контекст применения; ожидаемый пользовательский/бизнес-эффект; ограничения (данные, время ответа, стоимость, масштабируемость).
Минимум 1 baseline-решение, не использующее ML (или тривиальная ML-модель).
2. Первичный сбор данных и EDA
Описание источников данных и способов их получения.
EDA 
Анализ возможных утечек данных (data leakage)
3. Data Contract 
Для каждого источника данных:
схема данных;
обязательные поля;
допустимые диапазоны значений;
допустимая доля пропусков;
требования к актуальности данных (freshness);
ожидаемая периодичность обновления.
4. Архитектура системы 
Диаграмма архитектуры до выбора конкретных инструментов ( источники данных; обработка; хранение; модель; API / пользовательский интерфейс)
Заполненный ML System Design Doc
https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru
5. Риски (v0)
Минимум 5 рисков (данные; модель; инфраструктура; эксплуатация.) Для каждого риска: причина и возможные последствия.
Чекпоинт 2. Data Engineering и пайплайн данных
1. Data Pipeline
Выбор подхода ETL или ELT с обоснованием. Описание этапов пайплайна. Обоснование выбранных инструментов.
2. Feature Engineering
Описание набора признаков (источники; окна агрегации; частота обновления.)
Концепция Feature Store (registry признаков; разграничение offline / online признаков.)
3. Работа с данными
Использовать минимум один из подходов: семплирование; добавление данных; аугментация; исправление дисбаланса классов.
4. DFD (Data Flow Diagram)
Чекпоинт 3. Моделирование и эксперименты
1. Модели
Обучить минимум 2 модели разных классов
(например: линейная / бустинг / нейросеть / retrieval).
Сравнение с baseline.
2. Эксперименты (эксперименты с фиксацией данных, признаков, моделей и параметров)
3. Стратегия дообучения
Определение триггеров переобучения (дрейф данных; падение метрик; изменение входного распределения.)
План действий при срабатывании триггера.
Чекпоинт 4. Деплой, мониторинг и эксплуатация
1. Стратегия деплоя (Выбор стратегии и план отката (rollback))
2. Мониторинг. Мониторинг трёх уровней:
Данные (качество, дрейф, свежесть);
Модель (распределения предсказаний, уверенность);
Система (время ответа, ошибки).
Минимум 5 алертов (условие; причина; действие при срабатывании)
3. Runbook. Краткая инструкция «что делать, если алерт X сработал».
4. Демонстрация
Показ работающего приложения (на паре показать работоспособное приложение, чтобы я и студенты могли воспользоваться и тестировать)
Демонстрация реакции системы на деградацию
